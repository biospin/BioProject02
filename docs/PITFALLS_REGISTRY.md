## 상세 (표에 안 담기는 맥락)

### A2 — 회고적 사전등록: "봉인"의 진위는 시각이 가른다
법칙 held-out 검정에서 폐·위·두경부는 예측 커밋이 결과보다 앞섰지만(sealed-forward), 대장만 결과(afedc6a 04:45)가 예측(77c0633 05:06)보다 **21분 빨랐다**. 사전등록 표에도 대장만 "이미 관측"이라 적혀 있었다. 표 구조가 같다는 이유로 같은 형식의 LAW_TEST를 쓰면 **회고적 적합을 사전등록 증거로 세탁**하게 된다.
→ 대책: 같은 표 구조를 쓰되 상단에 RETROSPECTIVE 배너, "검정력 있는 봉인 확증" 집계에서 제외. **교훈: 사전등록 여부는 문서의 말이 아니라 커밋 시각으로 검증한다.**

### A3 — 양성대조 실패를 오독하면 멀쩡한 파이프라인을 버린다
위암 Lauren 0.5364는 언뜻 "임베딩·MIL이 고장"으로 읽힌다. 그러나 ① 같은 파이프라인의 MSI는 dev 0.899→holdout 0.860으로 정상 일반화 ② Lauren만 dev 0.963→holdout 0.536(−0.43) ③ pixel-mean 0.631이 MIL 0.536을 앞섬 ④ Lauren 유병률이 기관별 100%~18%. → 원인은 **Lauren 특이 site-교란**이고, site-disjoint 평가가 지름길을 정당하게 차단한 결과였다. **교훈: 실패는 "무엇이 실패했나"가 아니라 "무엇만 실패했나"로 좁혀야 원인이 나온다.**

### T2·T4·T5 — 셸 한 줄이 만드는 가짜 사실
셋 다 "명령이 성공했으니 사실"이라 믿어서 생겼다. exit code·프로세스 카운트·필드 번호는 **의미를 보장하지 않는다.** 특히 T2는 그대로 갔으면 "main에 이미 있으니 이관 불필요"라는 잘못된 결론으로 PR #32 자체가 안 열렸을 것이다. **교훈: 확인 명령의 출력이 내가 묻는 질문에 답하는지 한 번 더 본다.**

### C2·C3 — 병렬 세션 환경의 기본 실수
여러 창이 동시에 도는 환경에서 내 세션의 기억은 **최신이 아니다**. 착수 전 실측(파일 존재·커밋 로그·전 브랜치 grep)이 없으면 이미 끝난 일을 다시 하거나, 남이 잘하고 있는 걸 고치려 든다. **교훈: "내 기억"이 아니라 "현재 파일"이 근거다.**
