# 여섯 명이 한 서버에서 — 인프라·협업·자동화

> 한 줄 요약: 작업 서버가 A100 1대에서 A6000 3대로 교체되면서 접속·데이터 공유·형상관리 세 가지를 함께 다시 정비했고, 그 위에 팀의 일상 업무 알림을 자동화하는 OpenClaw 봇을 얹었다. 이번 편이 다루는 것은 그 성능을 떠받치는 사무실, 곧 인프라와 협업이다.

> 형식 메모: 본문은 접속·데이터 공유·형상관리에 자동화 한 축을 더한 네 갈래로, 각각을 사무실 이사에 빗대어 순서대로 짚는다.

앞선 편들이 슬라이드를 자르고, 숫자로 바꾸고, 그 숫자를 믿을 수 있는지 따지는 이야기였다면, 이 글은 그 모든 작업이 실제로 어디서 어떻게 벌어지는가에 대한 것이다. 여섯 사람이 한 기계 위에서 각자 다른 일을 하며 부딪히지 않으려면, 눈에 잘 안 띄는 바닥 규칙이 먼저 서 있어야 한다.

## 사무실이 이사 가면 함께 바뀌는 것

사무실이 이사를 가면 세 가지가 한꺼번에 바뀐다. 출입 방법이 달라지고(접속), 공용 창고와 개인 책상이 새로 나뉘고(데이터 공유), 여러 사람이 같은 문서를 고칠 때 그 수정을 어떻게 합치느냐(형상관리)가 문제가 된다. 이 프로젝트에서는 이 셋이 며칠 사이에 함께 정리 대상이 되었다. 작업 서버가 통째로 교체됐기 때문이다.

SpatialPathoAgent(BioProject02)는 여섯 명이 각자 역할(데이터, 임베딩, 모델링, 치료 근거, 비평, 오케스트레이션)을 맡아 한 서버 위에서 함께 작업하는 프로젝트다. 그 서버가 바뀌었다.

- 이전: A100 80GB 1장, 주소 `61.109.239.220`
- 현행: RTX A6000 49GB 3장, 32 vCPU, 503GiB RAM, 주소 `121.126.38.195`(내부망 `192.168.0.85`)

계산 자원으로 보면 큰 카드 한 장에서 중간 카드 세 장으로 옮겨간 셈이다. 그런데 실제 문제는 카드 성능보다, 이 기계가 바깥에서 곧장 닿지 않는 내부망 안으로 들어갔다는 데 있었다. 접속·공유·형상관리를 다시 세우지 않으면 누군가는 서버에 못 들어가고, 누군가는 남이 만든 임베딩을 찾지 못한다. 실제로 그런 일이 있었다. 하나씩 짚는다.

## 첫째 축: 직결에서 bastion 경유로 옮기기

옛 주소로 곧장 접속하던 방식(`61.109.239.220`에 바로 SSH)은 더 이상 통하지 않는다. 새 서버는 내부망에 있어서, 옛 주소를 이제 점프 호스트(bastion)로 거쳐 들어가야 한다. bastion은 바깥과 내부망 사이에 놓인 중간 경유지 서버다. 바깥에서 내부망 기계로 직접 갈 수 없으니 중간 문을 한 번 거친다.

```bash
ssh -J bastion@61.109.239.220 -p <port> <user>@192.168.0.85
```

포트(`<port>`)는 팀원마다 다르다. 이 접속법을 모르면 옛 방식으로 시도하다 계속 실패한다. 실제로 팀원 한 명(jamie)이 이 문제로 접속이 막혔고, 새 접속법을 따로 전달해야 했다. 접속이 막히는 원인이 늘 주소에 있는 것은 아니다. 기계가 내부망으로 들어가면 옛 직결 주소로는 닿지 않으니, 주소보다 경로를 먼저 봐야 한다.

같은 경유지 위에 협업용 공동 JupyterLab도 붙였다. 실시간 동시 편집이 되는 작업 환경으로, 이것 역시 bastion을 거치는 터널로 연다. 로컬 포트 8899를 서버 쪽으로 이어 준 뒤 브라우저로 접속한다.

```bash
ssh -L 8899:localhost:8899 -J bastion@61.109.239.220 -p <port> <user>@192.168.0.85
# 이후 브라우저에서 http://localhost:8899
```

참고로 이 서버(A6000 3장)는 모두의연구소(Modulabs)에서 무료로 제공한 것으로 파악된다(추정). 조건은 논문 Acknowledgments에 GPU 자원 제공처를 명시하는 것이다. 공짜로 빌린 계산 자원에도 갚아야 할 표기가 붙는 셈이라, 이 조건은 논문 단계까지 기억해 둔다.

## 둘째 축: 안 보이는 /home, 공유되는 /workspace

새 환경에서 가장 혼동이 컸던 지점이다. 각 팀원의 SSH 계정은 사실 별도의 Docker 컨테이너다(브리지망 `172.18.0.x`, hostname은 본인 계정명). 컨테이너란 한 기계 안에서 서로 격리된 작은 방 여러 개를 만드는 기술이다. 문제는 각자의 홈 디렉터리 `/home/<user>`가 그 방의 로컬 디스크(`/dev/sdb1`)라서, 다른 계정에서는 아예 보이지 않는다는 데 있다. 권한을 열어 줘도 소용이 없다. 마운트 자체가 없으니, 옆방 책상 서랍은 열쇠를 줘도 애초에 그 방에 존재하지 않는 셈이다.

컨테이너들 사이에서 공유되는 유일한 볼륨은 `/workspace`(`/dev/sda2`) 하나뿐이다. 그래서 규칙을 다음과 같이 확정했다.

- 공유 대상 데이터(임베딩, manifest, split, label)는 반드시 `/workspace/data/cache/biop02/`에 실제 파일로 둔다. 홈 경로에 두거나 홈을 가리키는 심링크로 우회하지 않는다.
- manifest의 `embedding_path`는 `/workspace/...` 절대경로로 적는다(개인 홈 경로 금지).
- 폴더 이름은 `<model>_<version>/`(예: `uni_v1/`, `conch_v1/`, `exaone_v2/`)로 통일한다.
- 새 공유 폴더는 `chmod 2775`(setgid) + `chgrp project`로 만들어, 하위 파일이 그룹 `project`를 자동 상속하게 한다. setgid를 걸어 두면 그 폴더 안에 새로 생긴 파일이 만든 사람과 무관하게 같은 그룹을 물려받아, 다음 사람이 읽지 못하는 사고를 막는다.
- 단, raw `.svs` 원본은 각자 스트리밍으로 받아 임베딩을 뽑고 지우는 개인 자산이라, 컨테이너 로컬에 둬도 무방하다(팀 공유·영구 보관 대상이 아님).

성능 관련 메모도 하나 남긴다. `/workspace`와 `/data`는 공용 SATA SSD(447GB)지만, 개인 대용량은 `~/data`의 HDD(14.6TB, 회전식)에 있다. raw 슬라이드와 임베딩이 이 회전식 HDD에 있어 타일 읽기 입출력이 병목이 된다. 그래서 추출 스크립트들이 SSD 스테이징 같은 우회를 쓴다(이 이야기는 「임베딩 I/O 가속」 편에서 따로 다뤘다).

공유 폴더가 안 보인다는 문제 앞에서 처음 떠올린 원인은 권한이었다. 그러나 권한을 열어도 보이지 않았고, 진짜 원인은 컨테이너가 달라 마운트 자체가 없다는 데 있었다.

## 셋째 축: 오래 열린 PR #15를 rebase 대신 merge로 닫기

마지막은 여러 사람이 같은 문서를 고칠 때 생기는 문제다. 임베딩 작업 묶음이 담긴 PR(코드 변경 요청) #15가 오래 열린 채 막혀 있었다. 그동안 `main`(공식 본류)이 다른 PR들로 크게 앞서 나가면서, 팀 문서(CLAUDE.md 등)가 곳곳에서 겹쳤기 때문이다. 인수인계 메모에는 "문서 다중충돌 — rebase로 재구성 필요, 일단 중단해 둠"이라고 적혀 있었다.

먼저 충돌의 실제 크기를 쟀다. rebase(내 커밋들을 최신 본류 위에 한 줄씩 다시 얹는 방식)를 바로 돌리지 않고, 가짜 병합을 한 번 돌려 충돌 파일 목록만 뽑았다.

```
git merge --no-commit --no-ff origin/main
git diff --name-only --diff-filter=U   # 충돌 파일만 출력
```

결과는 `CLAUDE.md` 하나였다. 다중충돌의 정체는, rebase가 19개 커밋을 순서대로 재생하면서 같은 충돌을 여러 번 되풀이해 보여준 것이었다. 같은 파일 하나를 열아홉 번 마주친 데서 온 착시였다. 충돌이 많다는 인상을 그대로 믿지 않고 가짜 병합으로 규모부터 재 보니, 부풀려진 인상과 실제 크기가 이렇게 갈렸다.

그래서 rebase 대신 merge를 택했다. 판단 근거는 두 가지다.

- 이 저장소는 원래 merge 커밋 방식이다. 최근 머지된 PR들이 전부 merge 커밋으로 남아 있어, rebase로 역사를 한 줄로 펴는 방식을 저장소가 요구하지 않는다.
- 겹침이 많을 때 merge는 합치는 지점이 한 번뿐이라 안전하고, 강제 푸시(force-push)도 필요 없다.

충돌 내용도 사소했다. 팀원 표의 한 줄이었는데, 포트 번호는 같고 괄호 안 표기만 달랐다. 원인은 내 브랜치도 포트표를 고쳤는데 그사이 `main`이 이미 다른 PR로 같은 표를 정정해 둔 것, 곧 내 수정이 남이 해 둔 일과 겹친 중복이었다. 반대로 내 브랜치가 남긴 진짜 새 내용은 포트표가 아니라 팀 공유 데이터 경로 규칙(위 둘째 축의 규칙) 문단이었고, 이것은 `main`에 없었다.

그래서 한쪽을 버리지 않고 양쪽을 합치는 쪽으로 풀었다. 포트표는 main 것을 채택하고(이미 정답), 새 문단은 그대로 살렸다. 양쪽이 모두 건드린 `AGENTS.md`는 git이 충돌 없이 자동 병합했지만, 충돌 없음이 내용이 맞음을 보장하지는 않으므로 포트·bastion 관련 줄을 직접 확인해 모순이 없는지 점검했다.

```
9e58c9e Merge remote-tracking branch 'origin/main' ...   # 충돌 1건 해소
c1a56fe Merge pull request #15 ...                        # 본류에 머지
```

push 직후 PR은 `MERGEABLE / CLEAN` 상태가 됐고 그대로 머지됐다. 여기까지가 인프라 정리 작업(kkkim)이다. 아직 열려 있는 PR #18(CPTAC 임베딩, 데이터 담당자 리뷰 대기)과 PR #21(비평 교차리뷰, 총괄 검토 대기)은 각각 다른 사람의 확인이 필요해 이번에는 손대지 않았다.

## 넷째 축: 인프라 위에 얹은 OpenClaw 자동화

여기까지가 사무실을 다시 세운 이야기라면, 그다음은 그 사무실에서 사람들이 놓치는 일을 줄이는 이야기다. 이 자동화는 kkkim이 아니라 braveji(Orchestrator, BIOP02-85)의 작업이다. 인프라 정비 위에 얹힌 별개의 축이라 여기 함께 기록하되, 실제 작업은 braveji가 했다.

이 프로젝트는 각자 맡은 역할을 JIRA 이슈로 나눠 진행한다. 문제는 사람이 JIRA를 자주 열어 보지 않는다는 데 있다. 새로 할당된 일이나 마감이 임박한 일을 놓치기 쉽다. 그래서 사람 대신 지켜봐 주는 도구를 붙였다. 두 개념만 먼저 잡아 둔다.

- OpenClaw: JIRA를 주기적으로 확인하다가, 새 할당이나 기한 임박 같은 변화가 생기면 Slack으로 알려 주는 봇 게이트웨이. 감시하고 알림을 보내는 역할만 하고, 코드를 대신 짜 주지는 않는다. 실제 코딩은 팀원이 Claude Code나 Codex로 따로 한다.
- MCP(Model Context Protocol): 봇이 JIRA 같은 외부 서비스에 표준화된 방식으로 접속하도록 연결해 주는 규약. 여기서는 Atlassian이 제공하는 원격 MCP 서버에 봇을 등록해, 봇이 JIRA 이슈를 읽어 오게 했다.

정리하면 흐름은 이렇다.

```
JIRA(이슈 할당) ── OpenClaw 봇(변화 감지) ── Slack DM(팀원에게 알림) ── 팀원이 Claude Code/Codex로 실제 작업
```

목표는 단순하다. 매일 아침 9시(한국시간, 09:00 Asia/Seoul), 팀원에게 오늘 볼 이슈 다이제스트를 Slack DM으로 보낸다. 내용은 새로 할당됐거나 기한이 3일 내로 임박한 미완료 이슈다. 이렇게 하면 각자 JIRA를 뒤지지 않아도 놓치는 일이 줄어든다. 정해진 시각에 반복 실행되는 이 작업을 cron이라 한다. 대상 OpenClaw 버전은 2026.5.18이다.

설정 과정의 실제 난관은 대부분 CLI가 문서와 다르게 동작하는 지점이었다. 세 함정이 커밋에 그대로 남아 있다.

### 함정 하나: 바뀐 서브커맨드 이름

처음엔 단수형 `openclaw agent ...`로 시도했지만 동작하지 않았다. 실제 CLI는 복수형 `openclaw agents ...`였다(commit b64bf33). 있을 거라 가정한 `agents show` 같은 하위 명령도 실제로는 없어, 눈앞의 CLI 기준으로 절차를 고쳤다(commit 2025cae).

### 함정 둘: 없는 옵션과 붙지 않는 Claude 모델

에이전트를 만들 때 하니스를 지정하는 `--harness` 옵션을 쓰려 했으나, 그런 옵션은 없었다. 대신 `--model`로 모델을 직접 지정하는 방식이었다(commit 52ee5a0). 그리고 처음엔 Claude 계열(`claude-sonnet-4-6`)을 붙이려다, 게이트웨이가 Claude에 연결하지 못한다는 걸 확인하고 `codex`로 전환했다(commit b09ceab). 봇이 판단을 내리려면 언어 모델이 필요한데, 붙일 수 있는 모델이 정해져 있었다는 뜻이다.

### 함정 셋: 로그인이 아닌 등록 방식

Atlassian MCP를 붙일 때 `login` 서브커맨드를 찾았지만 존재하지 않았다. OpenClaw의 `mcp`는 로그인하는 게 아니라 서버 주소를 등록만 하고, 인증은 실제 호출 시점에 런타임 OAuth로 처리하는 구조였다. 연결 방식도 까다로워서, SSE 엔드포인트(`/v1/sse`)나 `npx` 방식은 동작하지 않고 반드시 `streamable-http` + `/v1/mcp` 조합이어야 연결됐다. 연결법을 아는 것과 실제로 연결되는 조합을 찾는 것은 서로 다른 일이었다.

그 밖의 실무 팁도 런북에 함께 남겼다. 게이트웨이 데몬을 설치할 수 없는 공유 서버에서는 `tmux`로 백그라운드 실행해 세션을 분리해 둔다. `nvm: command not found`, `NPM_CONFIG_PREFIX` 충돌 같은 Node 환경 문제는 해결 절차를 나란히 적어, 다음 사람이 다시 헤매지 않게 했다.

현재 상태는 있는 그대로 적는다. Slack 봇과 MCP 등록, cron 절차까지 골격은 갖췄으나, 모델 provider 인증(키) 설정이 아직 남아 자동 다이제스트가 완전 가동되기 전 단계다. 남은 일은 provider 키를 채워 하루치 실제 전송을 검증하고, 그 뒤 팀원 전체로 cron을 확장하는 것이다. 아직 완성이 아니라 골격이라는 구분은, 가동되는 것처럼 부풀리지 않고 그대로 남긴다.

## 배운 점

- 접속이 안 된다는 말의 절반은 사실 경로 문제다. 기계가 내부망으로 옮겨지면 옛 직결 주소로는 닿지 않고, bastion 경유라는 새 경로를 알아야 한다.
- 공유 폴더의 전제부터 의심한다. 권한을 열었는데 왜 안 보이지의 답은 컨테이너가 달라 마운트 자체가 없다였다. 그래서 규칙을 `/workspace` 실파일로 확정하고 홈 심링크를 금지했다.
- 충돌이 많다는 인상을 그대로 믿지 않는다. 실제 규모는 가짜 병합으로 짧은 시간에 확인되고, 이번에는 파일 하나였다. 다중처럼 보인 것은 rebase의 반복 재생 탓이었다.
- 지시받은 명령보다 그 목적을 본다. rebase 재구성의 목적은 PR을 머지 가능하게였고, 저장소 관행상 merge가 더 안전했다. 방법을 바꾸되 이유를 먼저 밝혔다.
- 자동화도 눈앞의 도구가 진실이다. CLI 서브커맨드 이름, 붙일 수 있는 모델, MCP 연결 방식 세 곳 모두 문서·가정과 달랐고, 커밋에 근거를 남겨 다음 사람이 같은 길을 다시 헤매지 않게 했다.

## 남는 것

며칠 사이 접속·공유·형상관리 세 가지를 함께 정비했다. 접속은 bastion 경유로 통일했고(`ssh -J bastion@61.109.239.220 -p <port> <user>@192.168.0.85`), 데이터 공유는 컨테이너 격리를 전제로 `/workspace` 실파일 규칙으로 확정했으며, 오래 막혀 있던 PR #15는 rebase 대신 merge로 본류에 들어갔다(`c1a56fe`, `9e58c9e`). 그 위에 braveji가 OpenClaw 봇으로 JIRA-Slack 알림 자동화의 골격을 세웠고, 모델 provider 키 설정과 하루치 전송 검증이 남아 아직 완전 가동 전이다. 눈에 띄는 작업은 아니지만, 여섯 명이 한 서버에서 부딪히지 않고 일하기 위한 바닥이다. 이 바닥 위에서 앞선 편들의 타일링·임베딩·모델링이 진행됐다.
